# Design Brief

VocabGlass UI redesign の目的と判断基準。迷ったらこの文書に戻る。
現状分析は [ui-audit.md](ui-audit.md)。

## Primary user

スマートグラス (Meta Ray-Ban) を持つ語学学習者。身の回りの実物を
きっかけに、対象言語の単語を集めて学びたい人。

利用場面は 2 つに分かれる:
- 外・移動中: グラスをかけて、目に入った物をその場でキャプチャする
- 家・すきま時間: 貯めたカードをスマホだけで見返す、復習する

この 2 場面は姿勢が違う。前者は「早く始めて、あとは声で」、後者は
「じっくり触って学ぶ」。UI はこの 2 つを混ぜない。

## Primary action

新しいカードを 確認・保存・復習 する。

分解すると:
1. セッションを始めてキャプチャする (カードが生まれる)
2. 生まれたカードを確認する (デモの見せ場でもある)
3. 貯まったカードを復習する (フラッシュカード)

アプリを開いた瞬間に 1 と 3 のどちらへも 1 タップで入れること。
それ以外の操作 (編集、削除、設定的なもの) は一段奥でよい。

## Design goal

1. 最短距離でグラスと接続し、セッションを開始できる。
   アプリ起動からセッション開始まで、判断を求める画面を挟まない。
   接続状態はアプリが把握して見せる。ユーザーに推測させない。
2. 保存したカードの確認と復習がしやすい動線がある。
   デッキ一覧と、そこからのフラッシュカード復習が主動線。
   閲覧・編集・削除は一覧から自然に届く。

非目標 (今回やらないこと):
- ビジュアルのブランディング作り込み。まずは構造と動線
- 語学学習アプリ的な機能拡張 (spec.md の out of scope に従う:
  フォルダ、間隔反復、認証、同期)
- 手動キャプチャ画面 (現 ContentView) の磨き込み。開発用として残す

## Principles

判断基準として使う。優先順位の高い順。

1. **主要操作がひと目で分かる**
   各画面に主ボタンは 1 つ。「今この画面ですることは何か」が
   開いた瞬間に分かる。並列のフル幅ボタンを積まない。

2. **情報量を抑える**
   内部状態の生ダンプ ("stream: streaming", "session state: …") を
   ユーザーに見せない。状態は要約して 1 行、詳細はエラー時だけ。
   カード表示も学習に要る 4 要素 (word / pronunciation / meaning /
   example) と写真だけ。

3. **Native iOS らしく振る舞う**
   標準コンポーネントと標準ジェスチャを使う。削除はスワイプ、
   編集は詳細画面から、一覧は List、進行中は ProgressView。
   独自 UI を発明するのは標準に無い操作 (フラッシュカードの
   reveal) だけ。

4. **状態変化が分かる**
   接続中 / セッション中 / 生成中 / 保存済み の遷移が視覚的に
   伝わる。とくに「声でキャプチャしたらカードが現れる」瞬間は
   このアプリの核なので、明確に見せる。エラーは出しっぱなしに
   せず、原因と次の一手を添える。

## Success criteria

Redesign 後にこうなっていれば成功:

- 起動 → セッション開始が 1 タップ (接続済みのとき)
- 起動 → 復習開始が 2 タップ以内
- セッション中、ユーザーが読む文字は状態 1 行 + 最新カードだけ
- 削除とテキスト編集が UI から可能 (backend は実装済み)
- どの画面もシステム標準の見た目から大きく逸脱しない

## Open questions

- ルート構成: TabView (Capture / Deck) か、1 画面 + ナビゲーションか。
  2 場面を混ぜない原則からは TabView が素直だが、画面数が少ない
  うちは過剰かもしれない。
- 現 ContentView (手動キャプチャ) の置き場所。開発用に残すなら
  どこからアクセスするか。
- フラッシュカードの並び順: 新しい順か、シャッフルか。
  FlashcardSession は並びを呼び出し側に任せる設計なので UI 側の決め。
